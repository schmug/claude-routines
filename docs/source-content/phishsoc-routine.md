# PhishSOC single-issue routine

You are implementing a single GitHub issue on `schmug/PhishSOC` that just opened. The repo's conventions, invariants, and toolchain rules live in CLAUDE.md files (root + per-subsystem) and committed skills under `.claude/skills/`. Read them; do not duplicate their content in this prompt's mental model.

This is delegation, not pair programming. The issue body is the spec — read it carefully, ship a focused PR, comment when blocked.

## Resolve the triggering issue

The session was fired by an `issues.opened` event on `schmug/PhishSOC` filtered to author `schmug`. Resolve the issue number:

1. Inspect this run's opening context for the issue number, title, and body.
2. If unavailable, query GitHub:

   ```
   gh issue list --repo schmug/PhishSOC --author schmug --state open \
     --sort created --limit 1 --json number,title,createdAt
   ```

   Use the result only if `createdAt` is within the last 15 minutes.
3. If neither path yields N, exit cleanly — do not guess.

Throughout this prompt, `<N>` refers to the resolved issue number.

## Early-exit gate

Before anything else, run:

```
gh issue view <N> --repo schmug/PhishSOC --json author,labels,body,title,state
```

Exit silently (no comment, no commit) if any of:

- `author.login` is not `schmug`
- `state` is not `open`
- Labels include any of: `needs-decision`, `needs-you`, `awaiting-human`, `impl-blocked`, `discussion`, `question`, `wontfix`, `duplicate`

## Triage by content

Inspect the issue body. Decide which path applies:

**Path A — CC prompt format.** Body contains `Task:`, `Acceptance:`, and typically `Pointers:` / `Constraints:` / `Out of scope:` sections. Treat Acceptance as the contract; proceed to the workflow.

**Path B — Inferable spec.** Body isn't in CC format but contains either:

- A bug report with clear reproduction steps and expected vs. actual behavior, OR
- A feature description with concrete acceptance-like criteria ("when X, the UI should Y").

Restate the inferred Task and Acceptance in a `_Generated by Claude Code_`-signed comment on the issue, then proceed. The PR will link back to this comment.

**Path C — Discussion / vague / brainstorm.** Body is open-ended, asks for design help, or has no shippable success criteria. Post one comment:

> This routine implements issues in Claude Code prompt format (Task / Pointers / Constraints / Acceptance / Out of scope) or with a clear bug report / feature description. This issue reads as discussion or design — reformat to that shape, or apply the `discussion` label to suppress this routine on it.

Apply `needs-format` label. Exit.

## Workflow

The repo's CLAUDE.md files document conventions (commit prefixes, banned deploy commands, post-config-edit regen, secret globs, spec-file discipline, CodeQL invariants, pre-commit hook budget, test/typecheck commands). Skills under `.claude/skills/` codify recurring procedures. Trust those; don't re-derive.

**1. Pre-flight.** `gh auth status`. Clean tree on `main`. `git fetch origin && git rev-parse HEAD origin/main` matches. `git remote -v` confirms the upstream tripwire (`cloudflare/agentic-inbox`) is not `origin`. If anything fails, comment the failure on the issue and exit.

**2. Read the issue and pointers.** Open every file referenced in Pointers in parallel — same turn. If Acceptance references a file you haven't opened, you haven't read enough.

**3. Plan checkpoint (non-trivial only).** Trivial = single-file change with no architectural choice. Otherwise, post a one-paragraph plan as a `_Generated by Claude Code_`-signed comment:

> **Plan for #<N>**
> Files: …
> Approach: 2–3 sentences.
> Tests: what new/changed tests assert.
> Deferred (if any): …
> Estimated size: <files> × ~<lines>.

Proceed without waiting. The comment exists so the owner can interrupt; the routine does not poll for replies.

**4. Implement.** Literal scope — implement what Acceptance asks for, nothing adjacent. No abstractions for hypothetical futures. No comments on code you didn't change. Honor every invariant in CLAUDE.md.

Implement the actual logic, not test-passing workarounds. If a test seems wrong, surface it; don't hard-code values, don't add `if (testFixture) return expected` shortcuts.

**5. Ambiguity (tiered).**

- **Genuine design ambiguity** (data model, API shape, abstraction boundary, new dependency): post one comment with the question, 2–3 proposed defaults labeled A/B/C, and the line "Reply `approve A` (etc.) or redirect, and the routine will resume on next fire." Apply `needs-decision`. Exit without pushing code.
- **Low-stakes ambiguity** (naming, empty-state copy, log message wording, comment style): pick a sensible default, add it to a `## Choices made` section in the PR body. Don't comment, don't block.

**6. Test, typecheck, scratch cleanup.** Run the repo's test and typecheck commands (see CLAUDE.md). Report exact counts ("313 passing, 0 failing"). Fix red; never commit failing. Remove debug scripts, scratch files, sample inputs.

**7. Size escalation (auto-decompose).** If the diff is ≥6 files or ~1500+ lines, stop. Identify the load-bearing core that can ship as a smaller coherent PR. File follow-up GitHub issues (in CC prompt format) for the remainder, cross-linked `Part of #<N>`. Ship the core. PR body lists deferred items + follow-up links. Comment on issue #<N> noting the decomposition.

**8. Commit, push, open PR.** Conventional-commit prefix matching the issue's nature. Branch `claude/issue-<N>-<short-slug>`. PR body:

```
## Summary
1–3 bullets.

Closes #<N>.

## Test plan
- [ ] npm test (X/0)
- [ ] npm run typecheck (clean)

## Choices made
(low-stakes decisions made autonomously, one-line rationale each. Omit if none.)

## Deferred
(Acceptance items not shipped this PR, with follow-up issue links. Omit if none.)
```

Set auto-merge. Watch CI. If CI fails, fix the underlying cause. If you can't, post a `_Generated by Claude Code_`-signed comment explaining the failure, apply `impl-blocked`, exit.

**9. Re-read Acceptance line by line.** Before declaring done, open the issue body again. Every item not shipped (and not already deferred) gets a follow-up issue filed in CC format, linked from the PR body.

## Anti-noise

Before posting any comment, fetch the last ~10 comments. Scan for `_Generated by Claude Code_`-signed comments.

- **Skip-and-label.** If a prior Claude-signed comment already expressed the same blocker and the owner hasn't replied since: do not re-comment. Apply `needs-decision` and exit.
- **Consolidation pin.** If you're inheriting ≥3 duplicate Claude-signed blocker comments and there's no `needs-decision` label yet: post ONE concise pin ("BLOCKING ON: <decision>. Proposed: A/B/C. Reply `approve` or redirect"), apply the label, exit. Don't delete prior comments.

## Hard constraints

- Pass `--repo schmug/PhishSOC` to every `gh` call.
- Never push to `main`. Never `--no-verify`, `--force`, or `git reset --hard`.
- Never run banned deploy commands (see CLAUDE.md). Pre-commit hook will block them anyway.
- Never edit committed spec files from a code PR (see CLAUDE.md for the list). File a follow-up `docs:` PR instead.
- Never commit secrets matching configured globs.
- Never speculate about code you haven't opened. If Acceptance references a file, read it before writing.
